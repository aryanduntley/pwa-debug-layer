  So roughly 85% of the CDP-level surface is already done, and 0% of the   
  framework-introspection / replay / persistence layer that's actually your
   differentiator.                                                         
                  
  Recommendation: ship a focused companion MCP + extension, do not fork    
  
  The right shape is option (c) in your question. Concretely:              
                  
  1. Drop your planned CDP-overlapping tools (dom.* clicking/typing, nav.*,
   tabs.*, console.*, network.*, eval, perf.*). chrome-devtools-mcp does
  these well, is Google-maintained, gets free upstream improvements, and   
  uses the same AX-default snapshot model you specced. Re-implementing is
  pure cost.
  2. Build the companion MCP around what only an extension can do well:
    - react.tree, react.getState, react.findByText/findByRole              
    - vue.tree, vue.getState, svelte.*, solid.*                            
    - store.get/subscribe/dispatch (Redux/Zustand/Pinia/Jotai)             
    - events.tail over DOM + a small opt-in user-instrumentation API       
  (custom buses, RxJS)                                                     
    - session.record/replay (rrweb)                                        
    - Long-lived console.tail/network.tail across navigations (your plan's 
  ring buffers + disk spill) — keep these even though chrome-devtools-mcp  
  has per-nav versions, because the persistence semantic is materially
  different                                                                
    - Page-world instrumentation that auto-attaches on every page load
  (which initScript can't really replicate when the user's already on the  
  page)
  3. Why an extension still makes sense even though chrome-devtools-mcp can
   --autoConnect:                                                          
    - Extension hooks __REACT_DEVTOOLS_GLOBAL_HOOK__ etc. the moment the
  framework boots, on every page, automatically. Doing this through CDP    
  requires re-injecting via initScript and reloading.
    - Extension can monkey-patch fetch/XHR/WS from page-world cleanly via  
  world: "MAIN" content scripts — CDP can do it but it's clumsier and tied 
  to navigation lifecycle.
    - Extension naturally rides the user's real browsing session without   
  the WebDriver banner Puppeteer launches with.                            
  4. Do not modify chrome-devtools-mcp. Forking puts you on a maintenance
  treadmill against an upstream Google project, and your gap is orthogonal 
  to their layer (framework state lives above CDP).
  5. Two MCPs side-by-side is fine. Namespace yours                        
  mcp__pwa_debug__react_*, mcp__pwa_debug__store_*, etc. Claude Code       
  happily runs both; the user installs chrome-devtools-mcp for the CDP
  plumbing and pwa-debug-layer for framework + persistence + replay.
