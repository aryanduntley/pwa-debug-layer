  What changed in this session (the snap fix in two parts)
  1. Multi-kind install resolver
     packages/host/src/native-messaging/browser_paths.ts now exports
     detectBrowserInstalls(env, platform, exists) that returns
     BrowserInstall[] discriminated by kind:
       - 'native'   ~/.config/<browser>/NativeMessagingHosts/
       - 'snap'     ~/snap/chromium/common/chromium/NativeMessagingHosts/
       - 'flatpak'  ~/.var/app/<id>/config/<browser>/NativeMessagingHosts/
                    (with caveat about confinement; user may need
                    `flatpak override --user --filesystem=host <id>`)
       - 'registry' Windows HKCU\Software\<vendor>\NativeMessagingHosts\
                    com.pwa_debug.host (manifest JSON in %APPDATA%\pwa-debug)
     macOS uses the native variant under
       ~/Library/Application Support/<vendor>/NativeMessagingHosts/.

  2. Install-time launcher script (the snap-PATH fix)
     packages/host/src/native-messaging/launcher.ts emits a tiny POSIX sh
     launcher (or .bat on Windows) that hardcodes process.execPath +
     dist/main.js. The manifest's `path` field points at the launcher,
     never at main.js directly. Result: spawn succeeds even when the
     spawning browser's PATH is stripped (snap, flatpak).
     Linux launcher path: ~/.config/pwa-debug/bin/pwa-debug-host
     Windows launcher path: %APPDATA%\pwa-debug\pwa-debug-host.bat

  - manifest_writer.ts dispatches install/uninstall by install.kind,
