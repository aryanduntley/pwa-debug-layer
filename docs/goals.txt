A few honest updates that should change how you frame this:     This is happening, and the connector is MCP — not HTTP     Your bridge talks to me over curl because that's what was available when   you built it. The thing that's actually emerged for "AI sees and does  live" is MCP (Model Context Protocol). Claude Code calls MCP tools   natively — they show up the same way Read/Edit/Bash do. That means an AI  doesn't have to be told "curl this" — it just calls dom.query or  console.tail like any other tool, with structured results, no scraping.   Several MCP servers already do approximately what you described:     - chrome-devtools-mcp (Google, official) — connects to a running Chrome   and exposes CDP as MCP tools: live DOM, console, network, performance  traces, evaluate, click, type. This is the closest existing thing to your  vision.   - playwright-mcp (Microsoft, official) — same idea but  Playwright-flavored, with snapshot/AX-tree views.   - A handful of community CDP-MCP servers covering specific niches.    If you install one of those into Claude Code, the "I had to copy/paste   the DOM" friction largely disappears in this exact session — I can   DOM.querySelector and DOM.describeNode myself.     Where they fall short of your vision   The existing servers do CDP well but stop at the framework boundary. They  don't tell you:    - React component tree / state / props (what React DevTools shows)   - Vue/Svelte/Solid reactive graph  - Redux/Zustand/Pinia store state and dispatched actions   - Custom event buses or RxJS streams   - Service-worker activity beyond raw network     That's the genuine gap. The browser extension idea is the right shape for  filling it, because:     - chrome.debugger API gives an extension full CDP access (same powers as   Playwright, but on the user's real browser with their real sessions).  - Content scripts can hook the framework devtools globals   (__REACT_DEVTOOLS_GLOBAL_HOOK__, __VUE_DEVTOOLS_GLOBAL_HOOK__, the Redux   devtools extension hook) — which is exactly how the existing devtools  extensions work.   - An extension can monkey-patch fetch/XHR/WebSocket/postMessage from page  context for richer event capture than CDP gives you.   - It runs alongside the user's normal browsing instead of a  Playwright-spawned Chromium, which fits the "user is debugging their own   app" use case.     The tradeoffs to know up front: chrome.debugger shows a yellow "X is   debugging this browser" bar (annoying but harmless during dev), and the  Web Store has gotten stricter about extensions that exfiltrate page data   — for personal/dev use it's fine; for distribution there's policy work.   What I'd actually build, in order of effort     1. Install chrome-devtools-mcp first. Five minutes. Solves 70% of "AI can  see the page" today, in this session, with no new code. If that's  enough, stop.   2. If you hit the framework-state gap, build a small extension that:  - Acts as an MCP server (extensions can run a local WS that an MCP shim  wraps).   - Hooks the React/Vue devtools globals and exposes react.componentTree,  react.getState(componentId), etc.   - Mirrors console + network + unhandled errors into a ring buffer  Claude can tail.   - Exposes dom.query, dom.click, dom.type via chrome.debugger for the  action side.   3. For your existing BrowserAIComm, the same idea applies — replace the  curl HTTP surface with an MCP server wrapping the Playwright session.   Same primitives, AI-native delivery.
